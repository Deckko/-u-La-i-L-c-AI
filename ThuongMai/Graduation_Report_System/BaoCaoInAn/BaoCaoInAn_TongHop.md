## NHIỆM VỤ THỰC TẬP

Trong thời gian thực tập tốt nghiệp tại **Công ty Cổ phần SweetSoft**, học sinh thực tập được giao nhiệm vụ tìm hiểu quy trình phát triển phần mềm thực tế, khảo sát nghiệp vụ bán lẻ trực tuyến và xây dựng dự án cá nhân ứng dụng công nghệ thông tin.

Nhiệm vụ trọng tâm của đợt thực tập này là giúp học sinh tiếp cận môi trường làm việc chuyên nghiệp của doanh nghiệp phần mềm, củng cố và vận dụng các kiến thức lập trình Web (Next.js, Laravel API, Qdrant Vector Database) đã được trang bị tại nhà trường để xây dựng hoàn chỉnh hệ thống: **"Website bán hàng trực tuyến tích hợp AI Chatbot tư vấn khách hàng tự động và Semantic Search"**.

Các công việc cụ thể được giao bao gồm:
1. Tìm hiểu quy trình làm việc Agile/Scrum của các nhóm dự án tại doanh nghiệp.
2. Khảo sát nghiệp vụ quản trị kho, xử lý đơn đặt hàng và tương tác chăm sóc khách hàng.
3. Thiết kế cơ sở dữ liệu quan hệ PostgreSQL và cấu hình Qdrant Vector Database.
4. Lập trình giao diện Frontend (Next.js 16 App Router) hỗ trợ responsive đa thiết bị và song ngữ i18n.
5. Xây dựng APIs nghiệp vụ Backend (Laravel PHP) và tích hợp dịch vụ AI RAG Engine (FastAPI).
6. Triển khai xuất bản dự án tĩnh (Static HTML Export) lên dịch vụ Firebase Hosting của Google.
7. Thực hiện kiểm thử tích hợp hệ thống, rà soát và khắc phục các lỗ hổng bảo mật liên quan đến thông tin xác thực thô trong mã nguồn.

Học sinh thực tập có trách nhiệm thực hiện đúng tiến độ được phân công, ghi chép nhật ký thực tập đầy đủ mỗi tuần và hoàn thành cuốn báo cáo tốt nghiệp đạt chất lượng kỹ thuật theo quy định của nhà trường.

Trong bối cảnh đó, Thương mại điện tử (E-commerce) không còn đơn thuần là một kênh bán hàng phụ trợ, mà đã vươn lên thành huyết mạch sinh tồn của mọi doanh nghiệp bán lẻ. Tuy nhiên, cùng với tốc độ phát triển là sự thay đổi sâu sắc trong kỳ vọng của người tiêu dùng. Một website bán hàng tĩnh truyền thống với tốc độ tải trang chậm và khả năng tương tác nghèo nàn đã không còn đủ sức giữ chân khách hàng.

Qua khảo sát thực tế tại **Công ty Cổ phần SweetSoft**, em nhận thấy hai bài toán lớn chưa có lời giải tối ưu trên các website bán lẻ hiện hành:
Thứ nhất là sự hạn chế của công cụ tìm kiếm truyền thống. Hầu hết các hệ thống bán hàng hiện nay đều sử dụng phương pháp tìm kiếm thô (Keyword Matching). Cơ chế này chỉ so khớp chính xác từng ký tự chữ, khiến hệ thống hoàn toàn thất bại khi người dùng nhập sai lỗi chính tả, sử dụng từ đồng nghĩa hoặc nhập câu hỏi dạng mô tả ngôn ngữ giao tiếp tự nhiên.
Thứ hai là bài toán về dịch vụ chăm sóc khách hàng. Chi phí nhân sự trực chat rất cao nhưng không thể đảm bảo hỗ trợ trực tuyến 24/7, đặc biệt là vào giờ cao điểm hoặc đêm muộn, dẫn đến việc mất đi một lượng lớn khách hàng tiềm năng.

Xuất phát từ thực trạng đó, em đã tiến hành thực hiện đề tài tốt nghiệp: **"Thiết kế và xây dựng website trực tuyến tích hợp AI Chatbot và truy vấn ngữ nghĩa Semantic RAG"** cho thương hiệu thời trang cao cấp DECKKO. Hệ thống được phát triển dựa trên kiến trúc phân tách dịch vụ (Decoupled Architecture) kết hợp sức mạnh hiển thị tối ưu của **Next.js 16 App Router** ở Frontend, tính ổn định và an toàn giao dịch của **Laravel PHP API** kết hợp cơ sở dữ liệu **PostgreSQL** ở Backend, và dịch vụ **FastAPI Python** kết hợp cơ sở dữ liệu vector **Qdrant DB** để xử lý tìm kiếm ngữ nghĩa thông minh.

# PHẦN 2: NỘI DUNG

## CHƯƠNG 1: CƠ SỞ LÝ LUẬN VỀ KIẾN TRÚC PHẦN MỀM VÀ CÔNG NGHỆ TRÍ TUỆ NHÂN TẠO

### 1.1 Tổng quan về kiến trúc Web decoupled và APIs RESTful
Kiến trúc phần mềm phân tách (Decoupled Architecture hoặc Headless Architecture) là mô hình thiết kế hệ thống trong đó lớp hiển thị giao diện (Frontend) và lớp xử lý logic nghiệp vụ máy chủ (Backend) được tách biệt hoàn toàn thành hai ứng dụng độc lập, giao tiếp với nhau thông qua mạng internet bằng giao thức HTTP/HTTPS và các định dạng dữ liệu chuẩn như JSON hoặc XML.

Trong các kiến trúc truyền thống dạng Monolithic (mã nguồn Frontend và Backend gộp chung trong một ứng dụng duy nhất), sự phụ thuộc chặt chẽ giữa mã nguồn hiển thị và logic dữ liệu làm tăng độ phức tạp khi hệ thống phình to, đồng thời gây khó khăn lớn cho việc mở rộng quy mô (Scale). Khi Frontend và Backend bị ràng buộc, mọi thay đổi nhỏ ở giao diện đều yêu cầu biên dịch và triển khai lại toàn bộ hệ thống máy chủ, gây rủi ro gián đoạn dịch vụ rất cao.

Kiến trúc Decoupled giải quyết triệt để các hạn chế trên nhờ cơ chế phân định trách nhiệm rõ ràng (Separation of Concerns). Backend tập trung toàn bộ tài nguyên vào việc thực thi nghiệp vụ dữ liệu, bảo mật và lưu trữ cơ sở dữ liệu. Nó đóng vai trò là một kho cung cấp tài nguyên thuần túy, xuất bản dữ liệu ra bên ngoài dưới dạng các API RESTful (Representational State Transfer). REST là một phong cách kiến trúc phần mềm dựa trên các ràng buộc tiêu chuẩn, sử dụng các phương thức HTTP cơ bản như GET (đọc dữ liệu), POST (tạo mới dữ liệu), PUT/PATCH (cập nhật dữ liệu), và DELETE (xóa dữ liệu). Mỗi tài nguyên trên hệ thống được định danh bằng một đường dẫn URL (Uniform Resource Locator) duy nhất.

Frontend hoạt động hoàn toàn độc lập như một khách hàng (Client) tiêu thụ dịch vụ API. Nhờ sự phân tách này, nhóm phát triển Frontend có thể tự do lựa chọn các thư viện hiển thị hiện đại nhất mà không bị bó buộc bởi ngôn ngữ lập trình của Backend. Đồng thời, một hệ thống Backend API chuẩn hóa có thể phục vụ đồng thời cho nhiều nền tảng Frontend khác nhau như Website, ứng dụng di động (iOS/Android), thiết bị IoT hoặc các chatbot tích hợp của bên thứ ba mà không cần viết lại mã nguồn xử lý dữ liệu lõi. Điều này tối ưu hóa chi phí phát triển và mang lại tính linh hoạt cực kỳ cao cho doanh nghiệp.

### 1.2 Lý thuyết phát triển Frontend bằng React & Next.js 16
React là một thư viện JavaScript mã nguồn mở được phát triển bởi Meta (Facebook) vào năm 2013, chuyên dùng để xây dựng giao diện người dùng tương tác dạng Single Page Application (SPA). Nguyên lý thiết kế cốt lõi của React dựa trên kiến trúc hướng Component (Component-Based Architecture), chia cắt toàn bộ giao diện thành các thành phần giao diện nhỏ, độc lập và có khả năng tái sử dụng cao. React quản lý hiển thị thông qua cơ chế Virtual DOM (DOM ảo) - một bản sao nhẹ của DOM vật lý trong bộ nhớ. Khi trạng thái (State) của dữ liệu thay đổi, React sẽ tính toán sự khác biệt giữa DOM ảo mới và cũ (thuật toán Reconciliation) để chỉ cập nhật những phần tử thực sự thay đổi trên DOM vật lý, giúp tối ưu hóa hiệu năng render vượt trội so với các phương pháp cập nhật giao diện truyền thống.

Next.js là một framework React cấp tiến được phát triển bởi Vercel, giúp giải quyết các nút thắt cố hữu của các ứng dụng React SPA truyền thống như: hiệu năng tải trang ban đầu chậm và khả năng tối ưu hóa công cụ tìm kiếm (SEO) kém do trình duyệt phải tải toàn bộ file bundle JS lớn về rồi mới tự render giao diện phía Client (Client-Side Rendering - CSR).

Next.js 16 mang đến kiến trúc App Router đột phá dựa trên nền tảng React Server Components (RSC). Trong mô hình này, các component mặc định sẽ được render trực tiếp trên máy chủ (Server-Side Rendering - SSR hoặc Static Site Generation - SSG). Khi người dùng truy cập trang web, máy chủ sẽ xử lý dữ liệu và trả về trực tiếp tài liệu HTML tĩnh đã hoàn chỉnh cấu trúc. Trình duyệt chỉ cần hiển thị HTML này lập tức giúp người dùng có được nội dung xem dưới 1 giây. Các công cụ tìm kiếm (Googlebot, Bingbot) có thể dễ dàng đọc và đánh chỉ mục (Index) nội dung trang web ngay lập tức, giải quyết triệt để bài toán SEO cho các hệ thống E-commerce.

Next.js 16 cũng tích hợp cơ chế Hydration thông minh, sau khi HTML tĩnh được render, nó sẽ tải một lượng tối giản JavaScript cần thiết để kích hoạt các tương tác động của Client Server mà không làm ảnh hưởng đến tốc độ tải trang ban đầu. Cơ chế định tuyến dựa trên cấu trúc thư mục (File-system Routing) giúp việc phát triển các route động (Dynamic Routes) như trang chi tiết sản phẩm `/products/[slug]` trở nên vô cùng trực quan và dễ dàng quản lý.

### 1.3 Cơ chế xử lý máy chủ Backend bằng Laravel Framework PHP
Laravel là một framework phát triển ứng dụng Web mã nguồn mở viết bằng ngôn ngữ PHP, hoạt động theo mô hình kiến trúc MVC (Model-View-Controller). Kể từ khi ra đời, Laravel đã nhanh chóng trở thành framework PHP phổ biến nhất nhờ triết lý thiết kế hướng tới sự rõ ràng, dễ bảo trì và tích hợp sẵn hàng loạt các thư viện hỗ trợ xử lý bảo mật, transaction và giao tiếp cơ sở dữ liệu mạnh mẽ.

Trong kiến trúc của một hệ thống Backend API decoupled, Laravel đóng vai trò là lớp xử lý trung tâm (Core Controller). Toàn bộ các yêu cầu HTTP từ Client gửi lên đều được định tuyến qua hệ thống Route Middleware để thực hiện kiểm tra quyền truy cập (Authentication/Authorization) và ngăn chặn các cuộc tấn công phổ biến như SQL Injection, Cross-Site Scripting (XSS), hay Cross-Site Request Forgery (CSRF).

Điểm mạnh vượt trội của Laravel nằm ở Eloquent ORM (Object-Relational Mapping) - một trình ánh xạ cơ sở dữ liệu quan hệ sang lập trình hướng đối tượng cực kỳ trực quan. Eloquent giúp các lập trình viên tương tác với các bảng của PostgreSQL thông qua các class Model PHP mà không cần viết các câu lệnh SQL thủ công phức tạp. Eloquent cũng hỗ trợ thiết lập các mối quan hệ quan trọng như một-nhiều (One-to-Many) giữa sản phẩm và các biến thể, hay nhiều-nhiều (Many-to-Many) giữa người dùng và vai trò phân quyền.

Đặc biệt, trong các bài toán giao dịch thương mại điện tử, Laravel cung cấp cơ chế DB Transaction vô cùng tin cậy. Khi khách hàng tiến hành thanh toán, hệ thống cần thực hiện đồng thời nhiều thao tác ghi vào cơ sở dữ liệu: trừ số lượng sản phẩm trong kho, tạo bản ghi hóa đơn mới, cập nhật điểm tích lũy và trừ tiền tài khoản. Nếu một trong các bước này gặp lỗi (ví dụ: máy chủ bị ngắt kết nối giữa chừng hoặc kho hết hàng), cơ chế Transaction sẽ tự động thực hiện Rollback (khôi phục toàn bộ trạng thái dữ liệu trước khi giao dịch bắt đầu), đảm bảo dữ liệu không bao giờ bị rơi vào trạng thái mất nhất quán hay sai lệch.

### 1.4 Công nghệ Cơ sở dữ liệu Vector Database và hệ thống Retrieval-Augmented Generation (RAG)
Vector Database (Cơ sở dữ liệu Vector) là hệ thống cơ sở dữ liệu chuyên biệt được thiết kế để lưu trữ, quản lý và truy vấn các điểm dữ liệu dưới dạng vector đa chiều (Vector Embeddings). Khác với các cơ sở dữ liệu quan hệ truyền thống chỉ lưu trữ dữ liệu dạng chuỗi, số hoặc boolean và tìm kiếm dựa trên so khớp chính xác ký tự thô, Vector Database hỗ trợ tìm kiếm dựa trên khoảng cách hình học giữa các vector trong không gian nhiều chiều, thể hiện độ tương đồng về mặt ngữ nghĩa (Semantic Search).

RAG (Retrieval-Augmented Generation) là một kỹ thuật tiên tiến trong công nghệ Trí tuệ Nhân tạo nhằm tối ưu hóa kết quả đầu ra của các Mô hình Ngôn ngữ Lớn (LLM) như GPT-4o, Claude 3.5. Bản chất của các mô hình LLM là được huấn luyện trên một tập dữ liệu tĩnh tại một thời điểm trong quá khứ. Do đó, chúng không có khả năng tự biết các thông tin nội bộ của doanh nghiệp (như lượng tồn kho thực tế, bảng giá mới cập nhật của DECKKO) hoặc sẽ xảy ra hiện tượng ảo giác (Hallucination - bịa đặt thông tin khi không có dữ liệu trả lời).

Quy trình hoạt động của RAG giải quyết bài toán này qua hai bước cốt lõi:
1. **Truy xuất (Retrieval)**: Khi khách hàng gửi câu hỏi lên chatbot, hệ thống sẽ chuyển câu hỏi đó thành một vector nhúng. Vector Database (Qdrant) sẽ tìm kiếm trong cơ sở dữ liệu các vector mô tả sản phẩm có độ tương đồng ngữ nghĩa cao nhất với câu hỏi của khách hàng.
2. **Tăng cường và Tạo lập (Augmented Generation)**: Hệ thống sẽ lấy thông tin chi tiết của các sản phẩm tìm được ở bước 1 làm ngữ cảnh đầu vào (Context), gộp chung với câu hỏi gốc của khách hàng thành một Prompt hoàn chỉnh gửi lên LLM. Mô hình LLM lúc này chỉ đóng vai trò là một bộ biên dịch ngôn ngữ tự nhiên, đọc hiểu ngữ cảnh được cung cấp để tạo ra câu trả lời chuẩn xác và trung thực 100% dựa trên dữ liệu sản phẩm có thật, hoàn toàn loại bỏ hiện tượng bịa đặt thông tin.

### 1.5 Phép đo độ tương đồng Cosine và cơ chế nhúng văn bản (Embedding)
Cơ chế nhúng văn bản (Text Embedding) là quá trình chuyển đổi một đoạn văn bản thô (một từ, một câu hoặc cả một đoạn văn) thành một chuỗi các con số thực trong không gian đa chiều (thường là 1536 chiều đối với mô hình `text-embedding-3-small` của OpenAI). Quá trình toán học này giúp giữ lại các mối quan hệ ngữ nghĩa của ngôn ngữ. Các từ hoặc câu có ý nghĩa tương đồng nhau trong thực tế (ví dụ: *"áo thun"* và *"áo phông"*) sẽ được chuyển đổi thành các vector nằm rất gần nhau trong không gian đa chiều này.

Để xác định xem hai đoạn văn bản (ví dụ: câu hỏi của khách hàng và mô tả tính năng của sản phẩm) có tương đồng ngữ nghĩa với nhau hay không, toán học sử dụng phép đo độ tương đồng Cosine (Cosine Similarity). Phép đo này tính góc Cosine giữa hai vector $\mathbf{A}$ và $\mathbf{B}$ trong không gian đa chiều. 

Công thức toán học của Cosine Similarity được định nghĩa là tích vô hướng của hai vector chia cho tích độ dài của chúng:

$$\text{Cosine Similarity} = \cos(\theta) = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \sqrt{\sum_{i=1}^{n} B_i^2}}$$

Giá trị của Cosine Similarity nằm trong khoảng $[-1, 1]$. Trong các ứng dụng thực tế về tìm kiếm ngữ nghĩa, giá trị này thường dao động từ $0$ (hoàn toàn không tương đồng) đến $1$ (hoàn toàn trùng khớp ngữ nghĩa). Ngưỡng giá trị so khớp trong hệ thống được định cấu hình động là `score_threshold = 0.72`. Chỉ những sản phẩm nào có điểm số tương đồng ngữ nghĩa vượt qua ngưỡng này mới được coi là kết quả phù hợp để đưa vào ngữ cảnh phản hồi của Chatbot AI.

## CHƯƠNG 3: TRIỂN KHAI GIẢI PHÁP THIẾT KẾ VÀ XÂY DỰNG MÃ NGUỒN HỆ THỐNG

### 3.1 Đặc tả kiến trúc phân cấp nguồn Frontend Next.js 16 App Router
Next.js 16 App Router hoạt động dựa trên cơ chế Server Component làm nền tảng hiển thị. Cấu trúc mã nguồn của lớp giao diện (Frontend) được thiết kế đồng bộ để đảm bảo tốc độ tải trang tối đa và khả năng tối ưu hóa công cụ tìm kiếm:

* **src/app/products/[slug]/page.tsx**: Trang chi tiết sản phẩm động. Next.js tự động gọi API lấy thông tin sản phẩm và render tĩnh tại Build Time (Static Site Generation - SSG).
* **src/components/atoms/ErrorBoundary.tsx**: Component bao bọc toàn bộ ứng dụng ở cấp độ root để bắt và khoanh vùng các lỗi runtime ở client.
* **src/components/organisms/AISidebar.tsx**: Bảng Chatbot AI trượt từ cạnh phải màn hình, quản lý logic gửi tin nhắn đến FastAPI và hiển thị kết quả tư vấn theo dạng markdown tương tác.

Đồng thời, tệp tin [ErrorBoundary.tsx](file:///c:/antigravity/ThuongMai/nextjs-frontend/src/components/atoms/ErrorBoundary.tsx) ngăn chặn sụp đổ màn hình trắng khi có lỗi biên dịch client:

```tsx
'use client';

import React from 'react';
import { AlertTriangle, RefreshCcw } from 'lucide-react';

interface Props {
  children: React.ReactNode;
  fallback?: React.ReactNode;
}

interface State {
  hasError: boolean;
  error?: Error;
}

export class ErrorBoundary extends React.Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    console.error('[ErrorBoundary]', error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      if (this.props.fallback) return this.props.fallback;
      return (
        <div className="min-h-[40vh] flex flex-col items-center justify-center gap-4 p-8 text-center">
          <div className="w-16 h-16 rounded-full bg-red-500/10 border border-red-500/30 flex items-center justify-center">
            <AlertTriangle className="w-8 h-8 text-red-400" />
          </div>
          <div className="space-y-2">
            <h2 className="text-lg font-black text-zinc-50 uppercase tracking-wider">Đã xảy ra lỗi</h2>
            <p className="text-sm text-zinc-500 max-w-sm">{this.state.error?.message || 'Vui lòng thử lại.'}</p>
          </div>
          <button onClick={() => this.setState({ hasError: false, error: undefined })}
                  className="flex items-center gap-2 bg-amber-500 hover:bg-amber-600 text-zinc-950 px-6 py-2.5 font-black text-xs uppercase tracking-widest rounded-sm">
            <RefreshCcw className="w-3.5 h-3.5" /> Thử lại
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

```php
$orderInfo = DB::transaction(function () use ($cartItems, $data, $userId) {
    $totalAmount = 0.0000;
    foreach ($cartItems as $item) {
        $variant = ProductVariant::lockForUpdate()->find($item['variant_id']);
        if (!$variant || $variant->stock_qty < $item['quantity']) {
            throw ValidationException::withMessages(['cart_items' => "Sản phẩm đã hết hàng trong kho."]);
        }
        $variant->decrement('stock_qty', $item['quantity']);
        $totalAmount += $variant->price * $item['quantity'];
    }
```
